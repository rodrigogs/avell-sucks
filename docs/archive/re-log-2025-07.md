> **Historical record.** This merges the three original RE notes (`initial-re.md`,

> the decompiled OEM app. The *validated, current* hardware knowledge now lives in

> Throughout, **"GamingCenter" / "Gaming Center" = the OEM Avell/Uniwill app** we
> reverse-engineered. **"AvellSucks" = our replacement.** A few first-pass guesses
> were later corrected on real hardware; those are called out inline.

## OEM stack (observed live)

Processes: `GamingCenter.exe` (main WPF/.NET Framework 4.6.1,
`C:\Program Files\OEM\GamingCenter\`), `GamingCenterTray.exe` (tray),
`LaunchServGM.exe` (Windows service `GamingCenter`), `XtuService.exe`
(`XTU3SERVICE`, Intel XTU integration), `OSDTpDetect.exe` /
`OOBEI2CTpOnOffDetect.exe` (startup helpers).

Startup: `GamingCenter` → `LaunchCtrlGM.exe -R`, plus the two detect helpers.

No OEM-owned TCP listener or obvious named pipe was found — the app talks through
**WMI/ACPI, the registry, Windows messages, and direct HID/native DLL calls**, not
HTTP/socket IPC.

Binary types (DIE): `GamingCenter.exe` PE32 .NET CLR 4.0, Authenticode-signed;
`GamingCenterTray.exe` .NET/WPF + `System.Management`; `ECIO.dll` PE64 native C++
VS2017; `HardwareAccess.dll` .NET CLR 2.0, Confuser 1.x-protected (Intel IronCity:
OcMailbox/MSR/TurboRatio/TDP). Decompiled with `ilspycmd` 10.1.0.8386 into
`C:\Users\rdp\RE-tools\gaming-center-decompiled`. PDB evidence:
`D:\Max\Source\GamingCenter\...`.

Registry: settings under `HKLM\SOFTWARE\OEM\GamingCenter\...` (`MyFan3`,
`RGBKeyboardView\Debug`, `LightBarOnOff`, fan mode/autosave, `Support` feature bits).

## EC control primitive (confirmed, still current)

WMI under `root\WMI`, class `AcpiTest_MULong`, instance `ACPI\PNP0C14\1_1`, method
`GetSetULong` (`oobeview/WMIEC.cs` → `WMIReadECRAM`/`WMIWriteECRAM`):

```text
read:   Addr = 1099511627776 + Addr        // 0x100_0000_0000 | addr (2^40 + addr)
write:  Data = (Value << 16) | Addr        // NO read flag
```

Values are single-byte (`& 0xFF`). Events via `AcpiTest_EventULong`. A BIOS/ODM
write path also exists (`AcpiODM_Demo`, `PNP0C14\2_0`, `GetUlongEx7`) — **high
risk, unused.** Cross-ref: `WmiEcBackend.PackRead/PackWrite`.

## Fan control (confirmed)

From `FanManagementPage.cs` / `ECSpec.cs`:

- `ADDR_MAFAN_CONTROL_BYTE = 1873` (0x751): `0x00` normal/auto, `0x40` boost,
  `0xA0` (160) custom-advanced, `0x81..0x85` (129..133) basic levels L1..L5.
- Custom PWM sources `0x743..0x747` (1859..1863) for L1..L5; `MaxPwm = 0x8C`.
- PWM defaults `ADDR_L*_PWM_DEFAULT_MYFAN2 = 1929..1933`.
- Capability/support: `0x782` (1922, BIOS OEM byte 2), `0x742` (1858, MyFanboost2
  bit), `0x741` (1857, AP OEM byte), `0x75D` (1885, trigger byte 2).

⚠️ **Correction:** `ECSpec.cs` also lists a legacy `0x80` (128) "user fan" base.
That is a *base*, not a live mode — the actual custom bytes are `0x81..0x85`.
`FanModeMap` intentionally has no 128 entry (an earlier note's "128 + level"
framing is the OEM's arithmetic, not a distinct mode).

## Power limits (confirmed — corrected from first pass)

`FanManagementPage2` (the newer path) writes PL1/PL2/PL4 to EC registers
**0x783/0x784/0x785 (1923/1924/1925)**, byte watts. OEM vocab:
`SHORT/EXTENDED_WINDOW_..._DESIGN_POWER_LIMIT`, `SetPL124Tau`.

⚠️ **Correction:** the first pass floated a "PL1/PL2/**Tau**" model and speculative
registers `0x77F..0x781`. Both were wrong. **Tau is an Intel XTU/MSR setting (OEM
`-id 66`), not an EC register** — the EC reads it as 0 (`WmiEcBackend` reports
`TauSeconds: 0`). The speculative `0x77F..0x781` neighborhood was replaced by the
confirmed `0x783..0x785`.

## RGB keyboard (OEM path — our impl is a parked stub)

ITE HID: VID `0x0489`, PID `0xCE40`, UsagePage `0xFF00`/`0xFF01`, usage `0x01`;
`HidD_SetFeature`/`GetFeature` (9-byte feature reports) + 65-byte output reports
for per-row color. Report IDs (`LM_ITE_RGB.cs`): `0x08` effect, `0x12` load
picture, `0x14` per-index color, `0x16` row index, `0x88` read effect, `0x80`
firmware version. `LM_Manager` tries `LM_ITE_RGB` then `LM_EC_RGB`; KB types from
UsagePage + registry `KBTypeID` (65298 FourZone, 65282 MEZone_1st, 65283
MEZone_2nd). There's also an OEM-service CLI path (`ledkb /setdata 0x08 ...`,
`RGBLB /DUMP`) via `OemServiceWinApp.exe`.

Our RGB backend is **unfinished/untested** (the laptop's keyboard is physically
broken); it's parked in `app/_pending/rgb-hid/` and the UI ships the
`LocalRgbService` stub. See the top-level README's keyboard note.

## Read-only EC probe (2026-07-03 snapshot)

The dynamic-probe plan (correlate a single OEM-UI change with an EC read diff via
`scripts/read-ec-probe.ps1` / `ec-snapshot.ps1` / `run-dynamic-probe.ps1`) was
**FULFILLED** — its "do not write 1873 or 1859-1863 until approved" safety
boundary is now obsolete: writes shipped through `SafeEcWriter` + the allowlist,
verified on hardware.

⚠️ The raw probe read the control byte `1873` as `0x6CA0` and the PWM bytes as
16-bit WORDs (e.g. `0x8C78`). **Only the low byte is the real register value** —
these are point-in-time WORD-read evidence, not the register spec.

| Addr | Hex | (low byte) | Meaning |
|---:|---:|---:|---|
| 1857 | 0x741 | — | AP OEM byte (RGB/key event flags); also the disproven "Tau" lead |
| 1858 | 0x742 | — | support byte (MyFanboost2 bit 1) |
| 1859–1863 | 0x743–0x747 | — | custom fan PWM sources L1–L5 |
| 1873 | 0x751 | fan mode | main fan control byte |
| 1922 | 0x782 | — | BIOS OEM byte 2 (capability) |

## First-pass architecture guess (superseded)

⚠️ The first note recommended **.NET 10 + Avalonia + a Windows service/helper**,
with "one API process owns all hardware; the GUI never calls WMI/HID." **None of
that shipped.** AvellSucks is **WPF/.NET 10** (Windows-only), drives hardware
**in-process** via `AvellSucks.Core`/`.Core.Windows`, and the loopback ASP.NET
server is a separate *optional* tier — not the owner of hardware access. See
[`../ARCHITECTURE.md`](../ARCHITECTURE.md).

## Outstanding ambiguity noted at the time

Many OEM `catch {}` blocks swallow exceptions (driver error channels unknown);
`RGBLB /DUMP` offsets were inferred from fixed substring indices; EC write
high-word semantics are vendor-defined (we only know the app-level encoding).
